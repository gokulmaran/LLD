Q2. What is fork()? What does it return?
A: fork() creates a child process as an exact copy of the parent. It returns:

> 0 (child's PID) → in parent
= 0 → in child
< 0 → error

cpp#include <iostream>
#include <unistd.h>
using namespace std;

int main() {
    int x = 10;
    pid_t pid = fork();

    if (pid < 0) {
        cerr << "Fork failed!\n";
    }
    else if (pid == 0) {
        x = 20;
        cout << "Child:  x = " << x << ", PID = " << getpid() << "\n";
    }
    else {
        cout << "Parent: x = " << x << ", PID = " << getpid() << "\n";
        cout << "Child's PID seen by parent: " << pid << "\n";
    }
    return 0;
}

// Output:
// Parent: x = 10, PID = 1001
// Child:  x = 20, PID = 1002   ← separate copy of x!

Q4. What is exec()? How is it different from fork()?
A: exec() replaces the current process image with a new program. Unlike fork(), it does NOT create a new process.
cpp#include <unistd.h>
#include <iostream>
using namespace std;

int main() {
    cout << "Before exec\n";

    execl("/bin/ls", "ls", "-l", nullptr);  // replaces this process!

    cout << "After exec\n";  // ← NEVER printed!
    return 0;
}
// Output:
// Before exec
// (ls -l output here)

Q5. What is wait()? Why is it needed?
A: wait() makes the parent block until a child finishes. Without it, finished children become Zombie processes.
cpp#include <unistd.h>
#include <sys/wait.h>
#include <iostream>
using namespace std;

int main() {
    pid_t pid = fork();

    if (pid == 0) {
        cout << "Child working...\n";
        sleep(2);
        cout << "Child done.\n";
        exit(0);
    }
    else {
        cout << "Parent waiting...\n";
        int status;
        wait(&status);         // parent blocks here
        cout << "Child finished with status: "
             << WEXITSTATUS(status) << "\n";
    }
    return 0;
}

🕐 2. CPU Scheduling

Q8. What is CPU Scheduling? What are its goals?
A: CPU Scheduling decides which process gets the CPU when multiple are ready.
Goals:

Maximize CPU utilization, throughput
Minimize waiting time, turnaround time, response time

Key terms:
Arrival Time  (AT) = when process enters ready queue
Burst Time    (BT) = CPU time needed
Completion    (CT) = when process finishes
Turnaround    (TAT) = CT - AT
Waiting Time  (WT)  = TAT - BT
Response Time (RT)  = first CPU - AT
=============================================================
Q9. Explain FCFS with example. What is its drawback?
A: First Come First Served — processes scheduled in arrival order. Non-preemptive.
Process  AT  BT
  P1      0   6
  P2      1   4
  P3      2   2

Gantt Chart:
| P1 (0-6) | P2 (6-10) | P3 (10-12) |

TAT: P1=6, P2=9, P3=10   Avg TAT = 8.33
WT:  P1=0, P2=5, P3=8    Avg WT  = 4.33
Drawback → Convoy Effect: Short processes wait behind long ones.

Q10. Explain SJF. What is the difference between preemptive and non-preemptive SJF?
A: Shortest Job First — picks process with smallest burst time.
Process  AT  BT
  P1      0   7
  P2      2   4
  P3      4   1
  P4      5   4

Non-Preemptive SJF:
| P1(0-7) | P3(7-8) | P2(8-12) | P4(12-16) |
Avg WT = (0 + 6 + 3 + 7) / 4 = 4.0

Preemptive SJF (SRTF - Shortest Remaining Time First):
| P1(0-2) | P2(2-4) | P3(4-5) | P2(5-7) | P4(7-11) | P1(11-16) |
Avg WT = (9 + 1 + 0 + 2) / 4 = 3.0   ← always optimal!
Non-Preemptive SJFPreemptive SJF (SRTF)Running job interrupted?❌ No✅ Yes (if shorter arrives)Optimal?Not always✅ Minimum avg WT

Q11. Explain Round Robin. What is the effect of Time Quantum?
A: Each process gets a fixed time quantum (TQ). After TQ expires, it goes back to the ready queue.

Q12. Explain Priority Scheduling. What is Starvation? How to fix it?
A: Each process has a priority. CPU goes to highest priority process.
Process  AT  BT  Priority (lower = higher priority)
  P1      0   3      3
  P2      0   5      1   ← highest priority
  P3      0   1      4
  P4      0   2      2

Non-Preemptive:
| P2(0-5) | P4(5-7) | P1(7-10) | P3(10-11) |
Starvation: Low-priority processes never get CPU if high-priority keep arriving.
Fix → Aging: Gradually increase priority of waiting processes over time.
Every 1 minute waiting → priority += 1
Eventually low-priority process gets high enough to run

Q15. What is Paging? Explain with diagram.
A: Paging divides:

Physical memory → fixed-size frames
Logical memory → fixed-size pages

A Page Table maps logical pages → physical frames.
Logical Address:  [ Page Number | Offset ]
                        ↓
                   Page Table
                   Page 0 → Frame 3
                   Page 1 → Frame 7
                   Page 2 → Frame 1
                        ↓
Physical Address: [ Frame Number | Offset ]
Logical Address = 2050, Page Size = 1024
Page Number  = 2050 / 1024 = 2  → maps to Frame 1
Offset       = 2050 % 1024 = 2
Physical Add = (1 × 1024) + 2   = 1026

=============================================================================
Q16. What is a Page Fault? What happens when one occurs?
A: A page fault occurs when a process accesses a page not currently in RAM.
Steps on Page Fault:
1. CPU accesses virtual address
2. MMU checks page table → page not in RAM (invalid bit)
3. OS trap (page fault interrupt)
4. OS finds free frame (or evicts one using replacement policy)
5. Loads required page from disk into frame
6. Updates page table
7. Restarts the instruction
Page Fault Rate affects performance massively → called Thrashing if too frequent.

===============================================================================
Q17. What is Thrashing?
A: Thrashing occurs when a process spends more time paging than executing — too many page faults.
Cause:   Too many processes → each gets very few frames
Effect:  Constant page in/out → CPU utilization drops to near 0%

Fix:
1. Working Set Model — allocate frames based on locality
2. Reduce degree of multiprogramming
3. Add more RAM

🔒 4. Deadlock

Q20. What is Deadlock? What are its 4 necessary conditions?
A: Deadlock is a state where processes wait forever for resources held by each other.
Coffman's 4 Conditions (ALL must hold):
1. Mutual Exclusion   — Resource held by only one process at a time
2. Hold and Wait      — Process holds resource while waiting for another
3. No Preemption      — Resources can't be forcibly taken away
4. Circular Wait      — P1 waits for P2, P2 waits for P3, P3 waits for P1
cpp// Classic Deadlock Example
mutex A, B;

// Thread 1          // Thread 2
lock(A);             lock(B);
lock(B);  ←─────►   lock(A);   ← DEADLOCK!

======================================================

A:
                    Deadlock Handling
                          │
        ┌─────────────────┼──────────────────┐
    Prevention        Avoidance          Detection
   (break conditions) (safe state)     + Recovery

===========================================================
Q22. Explain Deadlock Prevention.
A: Break at least one of the 4 Coffman conditions:
ConditionPrevention StrategyMutual ExclusionUse sharable resources (read-only files)Hold and WaitRequest ALL resources at once before startingNo PreemptionAllow OS to forcibly take resourcesCircular WaitImpose ordering on resources (always lock A before B)
cpp// Preventing Circular Wait — always acquire in fixed order
void thread1() { lock(A); lock(B); /* work */ unlock(B); unlock(A); }
void thread2() { lock(A); lock(B); /* work */ unlock(B); unlock(A); }
//              ↑ same order → no circular wait!
========================================================================
Q23. Explain Banker's Algorithm (Deadlock Avoidance).
A: OS checks if granting a resource leads to a safe state before allocating.
Safe State = there exists a sequence where every process
             can finish using available + held resources

Example:
Processes: P0, P1, P2     Resources: A=10, B=5, C=7

         Allocation    Max Need    Need (Max-Alloc)   Available
         A  B  C       A  B  C       A  B  C          A  B  C
P0       0  1  0       7  5  3       7  4  3           3  3  2
P1       2  0  0       3  2  2       1  2  2
P2       3  0  2       9  0  2       6  0  0

Safe sequence: P1 → P0 → P2 ✅ (Safe State)
→ OS grants request

If no safe sequence exists → OS denies request!
===================================================================
Q24. Explain Deadlock Detection and Recovery.
A: Let deadlock occur, then detect and recover.
Detection: Use Resource Allocation Graph (RAG) or Wait-for Graph.

Cycle in RAG = deadlock (with single instance resources)

Resource Allocation Graph:
P1 ──► R1 ──► P2 ──► R2 ──► P1   ← CYCLE = DEADLOCK
Recovery Strategies:
1. Process Termination
   a) Kill ALL deadlocked processes (costly)
   b) Kill ONE at a time until deadlock resolved

2. Resource Preemption
   a) Rollback — take resource, roll process back to safe state
   b) Select victim with minimum cost
   c) Prevent starvation — don't always pick same victim

🔗 IPC — Inter-Process Communication

📌 PIPES

Q1. What is a Pipe? What are its limitations?
A: A pipe is a unidirectional communication channel between processes, using a producer-consumer model with a kernel buffer.
Parent Process                    Child Process
   write end ──────[kernel buffer]────► read end
   fd[1]                               fd[0]
Limitations:

Unidirectional only (half-duplex)
Works only between related processes (parent-child)
No persistence — data lost if not read
Fixed buffer size (typically 64KB on Linux)

cpp#include <unistd.h>
#include <iostream>
#include <cstring>
using namespace std;

int main() {
    int fd[2];
    pipe(fd);          // fd[0]=read end, fd[1]=write end

    pid_t pid = fork();

    if (pid == 0) {
        // Child: reads from pipe
        close(fd[1]);  // close unused write end
        char buf[64];
        read(fd[0], buf, sizeof(buf));
        cout << "Child received: " << buf << "\n";
        close(fd[0]);
    } else {
        // Parent: writes to pipe
        close(fd[0]);  // close unused read end
        const char* msg = "Hello from parent!";
        write(fd[1], msg, strlen(msg) + 1);
        close(fd[1]);
        wait(nullptr);
    }
    return 0;
}
// Output: Child received: Hello from parent!

Q3. What happens when you write to a pipe with no reader? What is SIGPIPE?
A: Writing to a pipe with no reader sends SIGPIPE signal to the writer. Default behavior is to terminate the process.
cpp#include <signal.h>

#include <iostream>

    cout << "SIGPIPE caught — reader closed!\n";
}

int main() {

    pipe(fd);
    close(fd[0]);   // close read end immediately

    char buf[] = "test";
    write(fd[1], buf, 4);  // ← triggers SIGPIPE
    return 0;
}

📌 SHARED MEMORY

Q4. What is Shared Memory? Why is it the fastest IPC?
A: Shared memory maps the same physical memory pages into multiple process address spaces — no data copying, no kernel involvement after setup.
Process A                Process B
┌──────────────┐         ┌──────────────┐
│ virtual addr │         │ virtual addr │
│  0x7fff1000  │         │  0x7fff8000  │
└──────┬───────┘         └──────┬───────┘
       │                        │
       └──────────┐  ┌──────────┘
              ┌───▼──▼────────┐
              │ Physical RAM  │  ← SAME PAGES
              │  (shared mem) │
              └───────────────┘

No copy needed — write once, other process reads directly!
cpp#include <sys/shm.h>
#include <iostream>
#include <cstring>
using namespace std;

int main() {
    // Create shared memory segment
    key_t key = ftok("/tmp", 65);
    int shmid = shmget(key, 1024, IPC_CREAT | 0666);

    // Attach to address space
    char* shm = (char*)shmat(shmid, nullptr, 0);

    pid_t pid = fork();

    if (pid == 0) {
        // Child: reads after parent writes
        sleep(1);
        cout << "Child reads: " << shm << "\n";
        shmdt(shm);
    } else {
        // Parent: writes to shared memory
        strcpy(shm, "Data via shared memory!");
        wait(nullptr);
        shmdt(shm);
        shmctl(shmid, IPC_RMID, nullptr);   // cleanup
    }
    return 0;
}
// Output: Child reads: Data via shared memory!
===================================================================================
Q5. What is the problem with Shared Memory? How do you solve it?
A: Shared memory has no built-in synchronization — race conditions occur without explicit locking.
cpp// PROBLEM: Race condition on shared memory
*shared_counter += 1;   // Thread A and B both do this
                        // Read → Increment → Write is NOT atomic!

// SOLUTION 1: Semaphore
#include <semaphore.h>
sem_t* sem = sem_open("/mysem", O_CREAT, 0666, 1);

sem_wait(sem);          // lock
*shared_counter += 1;   // critical section
sem_post(sem);          // unlock

// SOLUTION 2: POSIX mutex in shared memory
pthread_mutexattr_t attr;
pthread_mutexattr_setpshared(&attr, PTHREAD_PROCESS_SHARED);
pthread_mutex_t* mtx = (pthread_mutex_t*)shm_ptr;
pthread_mutex_init(mtx, &attr);

pthread_mutex_lock(mtx);
*shared_counter += 1;
pthread_mutex_unlock(mtx);

}

Q12. How do you handle signals in C++? What signals cannot be caught?
A: Use signal() or sigaction() to register a handler. SIGKILL and SIGSTOP cannot be caught, blocked, or ignored.
cpp#include <csignal>
#include <iostream>
#include <unistd.h>
using namespace std;

volatile sig_atomic_t running = 1;   // async-signal-safe type

void handle_sigint(int sig) {
    cout << "\nCaught SIGINT (" << sig << ") — shutting down\n";
    running = 0;
}

void handle_sigusr1(int) {
    cout << "SIGUSR1 received — reloading config\n";
}

int main() {
    // Using sigaction (preferred over signal())
    struct sigaction sa{};
    sa.sa_handler = handle_sigint;
    sigemptyset(&sa.sa_mask);
    sigaction(SIGINT, &sa, nullptr);   // Ctrl+C

    signal(SIGUSR1, handle_sigusr1);   // kill -SIGUSR1 <pid>

    cout << "PID: " << getpid() << " — running...\n";
    while (running) {
        sleep(1);
        cout << "tick\n";
    }
    cout << "Cleanup done. Bye!\n";
    return 0;
}

Q16. What is an Inode? What does it store?
A: An inode (index node) stores all metadata about a file except its name. The directory maps name → inode number.

                          │  [indirect ptr]      │

Q18. What are the types of File Allocation methods?
A: How does OS track which disk blocks belong to a file?
1. Contiguous Allocation:
File A: blocks [10, 11, 12, 13]   ← all contiguous
+ Fast sequential access
- External fragmentation
- Must know file size upfront
2. Linked Allocation:
Block 10 → Block 25 → Block 4 → Block 18 → NULL
+ No external fragmentation
- Random access is slow (must traverse chain)
- One corrupt pointer loses rest of file
3. Indexed Allocation (used by Unix/ext2):
Index block: [10, 25, 4, 18, ...]
               ↓   ↓  ↓   ↓
            data data data data
+ Fast random access
+ No external fragmentation
- Index block overhead
MethodSequentialRandomFragmentationContiguous✅ Fast✅ FastExternalLinked✅ OK❌ SlowNoneIndexed✅ OK✅ FastNone

Q19. What is Journaling in file systems? Why is it needed?
A: Journaling logs changes before applying them to disk — enables recovery after crash without full disk scan.
Without journaling (crash mid-write):
[Write metadata] ──✗ CRASH ──► inconsistent state
→ need fsck (full disk scan, very slow!)

With journaling:
[Write to journal] → [Commit] → [Apply to disk] → [Clear journal]
         ✗ CRASH here → replay journal on reboot (fast!)

Q21. What is Virtual File System (VFS)?
A: VFS is an abstraction layer in the kernel that provides a uniform interface — programs use the same open/read/write calls regardless of underlying file system.

Q22. What are File Permissions in Linux?
A: Linux uses a 3×3 permission model — owner / group / others × read / write / execute.
bash-rwxr-xr--  1  user  group  4096  file.txt
│└──┘└──┘└──┘
│ │   │   └── others: r-- (read only)
│ │   └─────── group: r-x (read + execute)
│ └─────────── owner: rwx (full)
└──────────── file type: - (regular), d (dir), l (link)

chmod 754 file.txt
# 7 = rwx (owner), 5 = r-x (group), 4 = r-- (others)
