Requirements of Instagram's Design
Learn about the requirements and computational needs for the Instagram service.
Functional requirements
  • Post photos and videos: Users can upload and share media publicly or with followers.
  • Follow and unfollow users: Users can follow/unfollow other accounts to curate their feed.
  • Like or dislike posts: Users can like or dislike posts of accounts they follow.
  • Search photos and videos: Users can search content based on captions, hashtags, and location.
  • Generate news feed: Users view a feed of posts from all accounts they follow.
  • Stories: 24-hour disappearing posts visible only to followers.
  • Direct messaging: Private media and text messages between users.
Non-functional requirements
  • Scalability: Support billions of users and millions of media uploads per day.
  • Availability: 99.99% uptime — users expect Instagram to be always accessible.
  • Durability: Uploaded media must never be lost; stored with multi-region redundancy.
  • Low latency: News feed generation under 200ms for optimal experience.
  • Eventual consistency: A post may not appear instantly to all followers but within seconds.

System Design: Instagram
Overview of the Instagram system design problem, scope, and approach.
What is Instagram?
Instagram is a photo/video sharing social platform with 1B+ users. Core features include a news feed, stories, reels, explore, and direct messaging. The main engineering challenge is storing and serving massive media at low latency while generating personalized feeds for hundreds of millions daily users.
Key challenges
  • Storing and CDN-serving petabytes of media globally with low latency
  • Generating personalized news feeds for 500M+ DAU in under 200ms
  • Handling celebrity post spikes (millions of reads triggered by a single write)
  • High availability and zero media data loss
Design of Instagram
High-level architecture — components, data flow, and the design diagram.
High-level architecture diagram
Component descriptions
Gateway API Gateway
Single entry point. Handles JWT auth, rate limiting, and routes requests to downstream microservices.
Core Upload Service
Receives media, assigns unique ID, stores raw file in blob storage, and publishes an event to the message queue for async processing.
Core Feed Service
Generates personalized feeds using a hybrid push/pull fanout strategy. Regular users get pre-computed feeds in Redis; celebrities use read-time merging.
Async Message Queue
Kafka or SQS decouples the upload path from media processing, feed fanout, and notifications — enabling independent scaling.
Async Media Processor
Consumes upload events, transcodes videos, generates thumbnails, pushes processed media to CDN, and updates metadata DB.
Storage Blob Storage + CDN
Immutable media stored in object storage (S3-compatible) with replication. CDN edge nodes serve >95% of media bytes globally.

System Design: Newsfeed System
Design a scalable newsfeed aggregating content from followed users.
A newsfeed aggregates content from users you follow and presents it in ranked or chronological order. Core challenges: fan-out at scale, low-latency feed reads, and ranking algorithms. Similar push/pull fanout strategy as Instagram's feed applies here.
System Design: Twitter
Design the core distributed components of Twitter.
Twitter serves 200M+ daily users with ~500M tweets/day. Core challenges: tweet fanout, trending topics, real-time search indexing, and timeline generation at extreme scale.
